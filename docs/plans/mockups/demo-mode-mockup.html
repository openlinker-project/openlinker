Implementation notes — what changed vs the pre-#1124/#1125 plan
Decision Rationale
POST /auth/demo-session removed Replaced by the standard POST /auth/register flow with auto-approve. #1125 ships full registration infrastructure — a bypass endpoint duplicates logic and bypasses the new status/approval model.
Role: viewer → read-only Shape B from docs/specs/product-spec-1123-rbac-depth.md settled on read-only as the canonical role name. viewer was pre-spec terminology; all references updated.
RegistrationService.register() checks DemoModeService.isDemoModeEnabled() When true: sets status: 'active', role: 'read-only' on the new user, and skips the pending queue entirely. When false: normal status: 'pending' flow from #1125 — no change. This is the single integration point: no new controller endpoint needed.
Register page callout driven by GET /system/config → demoMode Same server-side flag as the in-app banner. FE fetches once on startup with staleTime: Infinity. Callout and button copy adapt to demoMode — zero dead UI in production installs.
Demo banner condition: systemConfig.demoMode === true Shown to any authenticated user when demo mode is on — not gated on role. Admins exploring their own demo instance also see it (useful reminder). The read-only role itself is what disables write affordances (via usePermission()), not the banner.
Admin nav hidden for read-only users (existing) Shipped by #1124 — role-gated sidebar items. Demo users never see the Admin section. No additional demo-specific nav changes needed.
Login page demo bar + register link Replaces the old "Launch demo session" button that called POST /auth/demo-session. New flow: info bar sets context → "Create a free demo account →" navigates to /register. Simpler: reuses the registration page rather than a new endpoint.
