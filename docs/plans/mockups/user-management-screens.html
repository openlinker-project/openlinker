User Management — UI Mockups

Low-fidelity screens for review. Three surfaces: (1) self-registration page, (2) admin pending-approval queue, (3) full user-management list. All three share the cockpit token set and IBM Plex type pairing per docs/frontend-ui-style-guide.md.

Mirrors the existing /login guest layout. Shown when OL_REGISTRATION_ENABLED=true. When registration is disabled, the /register path returns HTTP 410 and the FE shows an appropriate message (not mocked here).

Design notes & open questions
  • Nav placement: "Users" appears under a "Platform" group in the sidebar nav, admin-only (guarded by usePermission('users:read') once that permission is wired).
  • Role options: The approve and role-change selects are populated from the server-known role list. When #1126 adds operator, it appears automatically with no FE change.
  • Self-protection: The backend rejects requests that would deactivate or demote the caller's own account. The UI pre-emptively disables those actions for the current session user.
  • Reject confirmation: Uses a Radix AlertDialog (not a browser confirm) to match the existing destructive-action pattern in the app.
  • Registration toggle: When OL_REGISTRATION_ENABLED=false (default), GET /register redirects to /login and the login page shows no "Create account" link.
  • Role change on active session: The role change takes effect on the user's next session (next JWT refresh). No forced logout from the admin side in v1.
  • Pagination: Page size 25, offset-based, matching existing patterns in the app.
  • Out of scope for this issue: Secret redaction (#1124), operator role (#1126), OL_DEMO_MODE (#1127).
