# Implementation Plan — #544 Seed real product images for the 5 OL-* PrestaShop dev fixtures

**Issue**: #544 `feat(dev): seed real product images for the 5 OL-* PrestaShop dev fixtures`
**Builds on**: #521 (the seeder itself, in `docker/prestashop/post-install-lib/30-seed-test-products.php`)

---

## 1. Goal

The dev-stack PS seeder creates 5 `OL-*` fixtures with names, EANs, references, prices, and stock, but no images — every product renders the generic placeholder camera icon in the PS admin and in any FE that consumes the dev shop. This breaks visual smoke testing of:

- The `storefrontBaseUrl` resolution code in `PrestashopProductMapper`.
- `ContentSuggestionService` reading product images for AI suggestion input.
- The Allegro offer-image upload path (the 5 OL-* fixtures are the only realistic test corpus).

**Layer**: DX / dev-stack only. No production code paths change. No backend or frontend src/ files touched.

---

## 2. Non-goals (explicit per the issue)

- **No AI-generated images.** Issue rejects this.
- **No PHP GD on-the-fly placeholders.** Issue rejects this.
- **No per-language image alts.**
- **No image upload via the PS WebService API in the seed path.** We write directly to the filesystem + DB (same code path the admin "upload image" UI uses).
- **No extension to non-`OL-*` fixtures.**
- **No brand-accurate photos** for fixtures #1 (Bosch) and #3 (Adidas). Generic substitutes (cordless drill / black tee) are fine — these are dev fixtures, not customer-visible listings, and finding permissive-licensed brand-specific photos is unrealistic.
- **No CC-BY / CC-BY-SA images.** Attribution surface is out of scope. CC0 / Public Domain or Pexels License only.

## 3. Research notes

### 3.1 Existing seed-script shape

`docker/prestashop/post-install-lib/30-seed-test-products.php` (353 lines):

- Idempotency gate at line 62–68: counts existing `OL-*` products, exits if ≥5.
- Boots `AdminKernel` at line 51–52 — already done because PS 9.x's `Product::delete()` cascade reaches `Image::deleteAutoGeneratedImage`, which dereferences `SymfonyContainer::getInstance()`. **Critical**: image-related code already requires the booted kernel today, so adding image-creation calls in the same try-block is safe.
- Existing helpers: `olGetOrCreateAttributeGroup`, `olGetOrCreateAttribute`, `olCreateBaseProduct`, `olCreateCombination`. Pattern: throw `RuntimeException` on `ObjectModel::add()` failure; the outer try/catch (line 338–352) does best-effort partial rollback by enumerating `OL-%` products and calling `Product::delete()` on each.
- `Product::delete()` cascade already cleans up `Image` rows + filesystem derivatives (this is exactly the behaviour the AdminKernel boot exists to support). So **the rollback path doesn't need extension** — it already handles images transitively. Verified by reading the existing comment at lines 19–24.

### 3.2 PrestaShop image-attach surface (PS 9.0.x)

The same code path the admin "upload image" UI uses:

```php
$image = new Image();

$image->position = Image::getHighestPosition((int) $product->id) + 1;
$image->cover = $isCover;

// ImageType derivatives (small_default, large_default, etc.):
$imageTypes = ImageType::getImagesTypes('products');
foreach ($imageTypes as $imageType) {
```

### 3.4 Image-byte sourcing strategy (the friction point)

The issue requires real photos under permissive licences (CC0 / Public Domain / Pexels). Two approaches:

**Approach A — Curate via research subagent + curl + jpegtran** *(chosen)*

0. **Network canary** (pre-flight): `curl -sIo /dev/null -L -w '%{http_code}' -A 'Mozilla/5.0' https://commons.wikimedia.org/`. Must return 200. (Pexels was probed and confirmed Cloudflare-blocked for HTML scraping — `www.pexels.com/photo/*` returns 403 to the subagent's fetch path, so Pexels is not usable as a discoverable source. Pexels CDN at `images.pexels.com` works for direct fetches of known URLs, so a Pexels image is acceptable only if the user manually supplies the photo ID + slug.)
1. Spawn a `general-purpose` Agent with a **tight** brief — accepted source: Wikimedia Commons only:
   - Source URL must match `https://commons.wikimedia.org/wiki/File:...`. The agent fetches that page (with `-A 'Mozilla/5.0'`), locates the licence template, and the file passes only if the page literally contains one of `cc-zero`, `PD-self`, `PDM-1.0`, `PD-shape`, `PD-textlogo`, or an explicit "Public domain" / "CC0" textual marker inside the licence box. Anything CC-BY / CC-BY-SA / GFDL is rejected. The direct image URL is the `og:image` `<meta>` value (which points to `https://upload.wikimedia.org/...`).
   - **No other hosts.** Reject Unsplash, Flickr, stock-photo CDNs, or arbitrary "image I found". Pexels is rejected for discoverability reasons (see step 0).
   - Per-candidate metadata required: `{ filename, sourcePageUrl, directImageUrl, author, licenseId, licenseTagVerbatim }` where `licenseTagVerbatim` is the literal substring grepped from the source page that proved the licence.
2. `curl -L -o ...` each direct URL, validate HTTP 200 + JPEG magic bytes (`FF D8 FF`).
3. `jpegtran -copy none -outfile clean.jpg dirty.jpg` — lossless EXIF/metadata strip (no quality re-encode). `jpegtran` is installed on this machine (`/opt/homebrew/bin/jpegtran`).
4. Validate dimensions + filesize + **post-strip metadata removal** via Python+Pillow:

```py
from PIL import Image
import os
img = Image.open(path)
img.verify()  # structural integrity
img = Image.open(path)  # verify() consumes the file, re-open for size check
assert max(img.size) >= 1000, f"{path}: long edge {max(img.size)} < 1000"
assert os.path.getsize(path) <= 300_000, f"{path}: {os.path.getsize(path)} > 300kB"
exif = Image.open(path).getexif()
assert not exif, f"{path}: stripped file still has EXIF: {dict(exif)}"
```

The EXIF assertion guarantees `jpegtran -copy none` actually did its job — Pillow `verify()` alone catches corruption but not metadata leakage.

5. Commit the 7 cleaned files + `LICENSES.md` provenance.

**Approach B — Document URLs only, defer fetch to user**

Not a fallback to ship; documented here only as a contingency framing. See R1 below.

**Decision**: Approach A. The research subagent gives me live web search constrained to two licence-verifiable hosts. `curl` + `jpegtran` + Pillow validation (including post-strip EXIF assertion) gives a fully self-contained PR.

**Risk**: I might end up with 7 photos that satisfy the constraints but don't visually match the fixtures (e.g. a photo of *a* cordless drill that isn't quite the Bosch GSR12V-15). The issue accepts this for branded fixtures; the spirit is "anything reasonable for that product category". Acceptable.

### 3.5 No automated tests for the seed script

The seeder is a dev-only artifact — no `*.spec.ts` covers it today. The issue specifies smoke checks documented in the PR description (visual inspection + storefront image-URL check). Match that bar.

## 4. Design

### 4.1 New helper — `olAttachProductImage`

Insert into `30-seed-test-products.php` after the existing `olCreateCombination` helper (line 218). Signature matches the issue spec:

```php
function olAttachProductImage(
    int $idProduct,
    ?int $idCombination,
    string $reference,
    bool $isCover,
    string $srcFilename, // basename inside seed-images/, e.g. 'OL-MUG-LIN-300.jpg'
): int {
    $src = '/tmp/post-install-lib/seed-images/' . $srcFilename;
    if (!is_readable($src)) {
        throw new RuntimeException("Source image not readable: {$src}");
    }
    if (!is_writable(_PS_PROD_IMG_DIR_)) {
        throw new RuntimeException("PS image dir not writable: " . _PS_PROD_IMG_DIR_);
    }

    $image = new Image();
    $image->id_product = $idProduct;
    $image->position = Image::getHighestPosition($idProduct) + 1;
    $image->cover = $isCover;
    if (!$image->add()) {
        throw new RuntimeException("Image::add failed for {$reference}");
    }

    $dst = _PS_PROD_IMG_DIR_ . $image->getImgPath() . '.jpg';
    if (!@copy($src, $dst)) {
        $image->delete();
        throw new RuntimeException("Image copy failed: {$src} → {$dst}");
    }
    if (!ImageManager::resize($dst, $dst)) {
        $image->delete();
        throw new RuntimeException("ImageManager::resize failed for {$dst}");
    }

    foreach (ImageType::getImagesTypes('products') as $imageType) {
        $derivative = _PS_PROD_IMG_DIR_ . $image->getImgPath()
            . '-' . $imageType['name'] . '.jpg';
        ImageManager::resize(
            $dst, $derivative,
            (int) $imageType['width'], (int) $imageType['height'],
        );
    }

    if ($idCombination !== null) {
        Db::getInstance()->insert(
            'product_attribute_image',
            [
                'id_image' => (int) $image->id,
                'id_product_attribute' => $idCombination,
            ],
            false, true, Db::INSERT_IGNORE,
        );
    }

    return (int) $image->id;
}
```

**Why pre-flight `is_readable` / `is_writable` checks?** Aligns with the existing seed-script convention of throw + rollback (lines 122, 149, 176, 201). Silent skips are a debugging nightmare.

**Why no `$idShop` parameter?** `Image::add()` inherits shop scope from `Context::getContext()->shop`, which is already set up by the AdminKernel boot at line 51–52. Unlike `StockAvailable::setQuantity` (which is genuinely shop-scoped and needs `$idShop` passed explicitly), `Image` rows take their `id_shop_default` from context. Carrying a dead parameter on the signature would invite a future maintainer to "fix" it incorrectly.

## 6. Risks & mitigations

### R1 — Research subagent can't find 7 qualifying images
**Likelihood**: low to medium. **Impact**: blocks Phase 4.
Wikimedia Commons has thousands of CC0 / PD photos for everyday objects (mugs, tees, soap, rings, drills). Pexels has explicit "Free for commercial use, no attribution required" on Medium downloads.

**Mitigation**: if the agent returns <7 valid candidates, spawn a second pass with widened search terms (e.g. "ceramic cup" instead of "stoneware mug"). If that still fails for a specific subject, **fail forward**: surface the gap to the user (which subjects, why, what was rejected, candidate URLs the user could approve manually) and pause. **Do not ship a partial PR.** Shipping the seed-script + LICENSES.md template + a fetch-helper without the image bytes would put a broken seeder on the branch (the new helper would throw `Source image not readable` for the missing files on every fresh dev-stack boot). The right move is either user-supplied image bytes or scope reduction (e.g. drop fixture-4 per-combination images and ship 5 covers only).

### R2 — `Image::getImgPath()` semantics differ in PS 9.x
**Likelihood**: low. **Impact**: image upload silently writes to wrong path.
PS 9.x kept the legacy `Image::getImgPath()` behaviour — returns the multi-level dir path derived from `id_image` (e.g. `9/4/3/943` for id 943). Verified by reading PS 9.0.2 source today.

**Mitigation**: the helper's `RuntimeException` on copy failure surfaces this immediately during the smoke test. If `Image::add()` succeeds but `copy()` writes to a non-existent dir, copy returns false → throw → rollback.

### R3 — `jpegtran -copy none` produces a truncated/invalid file
**Likelihood**: very low. **Impact**: PS rejects the image during `ImageManager::resize`.
`jpegtran` is the canonical tool for lossless JPEG metadata stripping; it's been stable for 20+ years.

**Mitigation**: post-strip Python+Pillow validation re-opens each file with `Image.open(path).verify()` to catch corruption before commit.

### R4 — Per-combination image not visible on storefront
**Likelihood**: low. **Impact**: dev-only — would mean fixture #4 doesn't actually exercise the per-combination image path it's meant to.
PS 9.x's `product_attribute_image` table is the canonical link; the front-office product page reads from it via `Combination::getById()` → `Image::getImages()`.

**Mitigation**: the smoke step (§5 step 10) explicitly checks "fixture 4 shows different images per combination" by visiting the storefront product page and toggling between Lavender and Rose. If the combo image doesn't swap, the SQL insert is broken — fix before commit.

### R5 — Total image weight inflates beyond 2 MB
**Likelihood**: low (if filtering by Pexels Medium / Wikimedia 1024px thumbnails). **Impact**: PR violates the issue's budget.

**Mitigation**: post-strip Python+Pillow validation enforces ≤300 KB per file (issue spec) and total ≤2 MB. If exceeded, re-fetch a smaller source variant (Wikimedia "800px" thumbnail) for that file.

