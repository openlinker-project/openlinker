# Demo Analytics Events

This is the marketing handoff table for the demo-mode PostHog event catalog. The
source of truth is `apps/web/src/features/demo/lib/demo-events.ts` — this
document is generated by hand from that file and must be kept in sync whenever
an event is added, renamed, or removed.

**Status**: framework only (#1786). This table has one seed event today; the
full set of viewer-mode events lands across #1788, #1789, and #1790.

Events only fire on a demo-mode build (`OL_DEMO_MODE=true`) with PostHog
configured (env vars or the `/settings` panel) **and** the visitor has
accepted the analytics-consent banner. On a normal self-hosted install, none
of this code path runs.

## Enabling event groups (#1787)

Product events are off by default even on a fully-configured PostHog
connection. An admin turns them on at `/settings` → PostHog → **Product
events**: a master toggle (independent of autocapture) plus one toggle per
event **group** — groups are derived automatically from `DemoEventCatalog`,
never hand-maintained, so a new group added to the catalog appears in the
panel with zero settings-page edits. `captureDemoEvent` is a no-op unless
both the master toggle and the specific event's group are enabled.

| Event | Description | Props | What it measures |
|---|---|---|---|
| `demo_viewer_locked_action_clicked` | Viewer clicked a locked (read-only) write action — the primary intent-to-convert signal for a read-only demo session | `actionName: string`, `surface: string` | How often, and where, a read-only demo visitor attempts the final write action that `ReadOnlyLock`/`useWriteAccess` renders as disabled — the highest-value conversion-intent signal available from a viewer-only session. |

## Adding a new event

1. Add an entry to `DemoEventCatalog` in `apps/web/src/features/demo/lib/demo-events.ts` with a `description`, a `group`, and a typed `props` shape (bounded strings/numbers/booleans only — never PII, free text, or entity ids). The `props` value must carry real placeholder values for every key (e.g. `{ actionName: '', surface: '' } as { actionName: string; surface: string }`), **not** a bare `{}` cast — the `/settings` Product-events panel introspects prop *names* via `Object.keys(entry.props)` at runtime, and a bare `{}` silently shows "props: (none)" in that panel. `demo-events.test.ts` asserts every entry has non-empty runtime prop keys.
2. Call `captureDemoEvent(eventName, props)` (exported from the `demo` feature barrel) at the call site.
3. Add a row to the table above.
